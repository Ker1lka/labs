# Лабораторна робота №9: Організація Software Configuration Management для навчального проєкту

**Тема:** Організація Software Configuration Management для навчального проєкту
**Проєкт:** Вебзастосунок на Spring Boot (Java 17, Maven, MySQL, Lombok)

---

## 📚 1. Теоретичні відомості про SCM

**Software Configuration Management (SCM)** — це комплексна інженерна практика управління життєвим циклом системи. Вона включає контроль за всіма змінами програмного продукту, куди входить не лише вихідний код, а й інфраструктурні скрипти, документація, версії бібліотек та налаштування деплою. 

Основна відмінність SCM від звичайного використання Git полягає в тому, що Git — це технічний інструмент (VCS), а SCM — це управлінська дисципліна, яка впроваджує правила розробки: простежуваність (traceability), фіксацію стабільних станів (baseline) та правила релізів (release management).

## 📊 2. Завдання 1. Визначення configuration items

У нашому Spring Boot проєкті було визначено 5 ключових елементів конфігурації (Configuration Items), які перебувають під жорстким аудитом та контролем версій:

| № | Configuration item | Призначення | Чому має бути під контролем SCM |
|---|---------------------|-------------|---------------------------------|
| 1 | `src/main/java/...` | Вихідний Java-код додатка (контролери, сутності, репозиторії). | Містить бізнес-логіку. Неконтрольовані зміни призведуть до появи багів та деградації функціоналу. |
| 2 | `pom.xml`           | Декларативний файл збірки Maven та опис усіх dependencies. | Визначає версії бібліотек (Spring Boot, MySQL Connector, Lombok). Зміна версій може викликати конфлікти сумісності. |
| 3 | `application.yml`   | Конфігурація середовища (налаштування портів, логерів, підключення до бази даних MySQL). | Зміна параметрів (наприклад, невірний URL до БД) унеможливить запуск додатка на сервері. |
| 4 | `.github/workflows/ci.yml` | Конфігурація автоматизованого пайплайну CI (GitHub Actions). | Будь-яка помилка в цьому файлі ламає автоматичну перевірку коду, тестування та деплой всього проєкту. |
| 5 | `README.md`         | Головна технічна документація та інструкція з розгортання проєкту. | Застаріла або невірна документація веде до помилок нових розробників при локальному запуску системи. |

## 🛠️ 3. Завдання 2 та 3. Організація контрольованої зміни та Traceability

У межах роботи було реалізовано контрольовану зміну: **інтегровано логування подій у `CompanyController`** для покращення моніторингу працездатності додатка.

Процес розробки пройшов повний цикл простежуваності (**Traceability**): від постановки задачі до автоматичної перевірки та злиття в головну гілку.

### Таблиця простежуваності зміни:

| Етап процесу SCM | Об'єкт / Ідентифікатор | Роль у ланцюжку зміни |
|---|---|---|
| **1. Ініціація (Issue)** | Задача на GitHub Issues | Фіксація потреби розробки: опис необхідності додавання `log.info` в контролер. |
| **2. Ізоляція (Branch)** | Гілка `feature/company-logging` | Створення окремого простору в Git для безпечної розробки фічі без впливу на стабільний код. |
| **3. Модифікація (Commit)** | Коміт з описом змін та прив'язкою до іссуе | Додавання в код анотації `@Slf4j` від Lombok та виклику логера. Коміт марковано номером задачі для автоматичного зв'язку. |
| **4. Автоматична перевірка (CI)** | Пайплайн `ci.yml` (GitHub Actions) | Запуск віртуального середовища на базі Ubuntu, встановлення JDK 17 (Temurin), запуск тестів `mvn clean test` та збірка проєкту. |
| **5. Контроль якості (PR)** | Pull Request на GitHub | Етап інспекції коду (Code Review) та перевірка успішності проходження хмарного пайплайну CI. |
| **6. Інтеграція (Merge)** | Злиття Pull Request в `main` | Офіційне включення перевіреного та протестованого коду в основну гілку продукту. |

## 📌 4. Завдання 4. Формування baseline

Після того, як Pull Request пройшов автоматичний пайплайн GitHub Actions і був успішно змерджений у гілку `main`, систему було зафіксовано у новому контрольному стані — **Baseline**.

* **Назва baseline:** `BASELINE-STAGE-V1.0.0`
* **Склад baseline:** Актуальний сирцевий код Java (з інтегрованим логуванням), стабільні конфігурації Maven (`pom.xml`), інфраструктурні файли пайплайнів та документація.
* **Обґрунтування стабільності:** Даний стан системи є повністю працездатним: код пройшов успішну компіляцію, автоматичні тести виконані без помилок (статус `BUILD SUCCESS`), конфігураційні файли відповідають стандартам розробки.
